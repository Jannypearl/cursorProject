# Reference

## 常用动词建议

- `增加`：新增功能、接口、配置项
- `修复`：错误修正、兼容性问题处理
- `优化`：性能、结构、可维护性改进
- `重构`：代码组织调整且行为基本不变
- `补充`：文档、日志、测试或说明完善

## 提交信息粒度建议

- 改动单一：一句话即可
- 涉及“行为变化 + 配套改动”：标题 + 补充两行
- 跨模块改动：标题 + 2-3 条要点，不超过 6 行正文

## 生成时建议参考的信息

1. 本次 staged/staged+unstaged 的主要差异
2. 最近 10-20 条提交信息风格
3. 是否包含用户明确目标（如“修复线上故障”）

## 避免事项

- 空泛描述：如“更新代码”“修复问题”
- 把多个独立主题塞进一条提交信息
- 声称未经验证的结果（例如“性能提升 50%”）
- 使用与仓库明显冲突的风格（如历史全中文而输出全英文长句）
